1. Introdução

Objetivo

Desenvolver um backend robusto e completo para um sistema de gestão de restaurantes, permitindo que múltiplos estabelecimentos gerenciem eficientemente suas operações através de uma plataforma única e compartilhada. A Fase 1 concentra-se na gestão de usuários com autenticação JWT e validações robustas.

Escopo

  • Gestão completa de usuários (CRUD)
  • Autenticação e autorização com JWT
  • Validação robusta de dados
  • Integração com PostgreSQL
  • Documentação com Swagger/OpenAPI
  • Suporte a Docker e Docker Compose

Requisitos Obrigatórios Atendidos

  • ✅ Cadastro, atualização e exclusão de usuários
  • ✅ Troca de senha em endpoint separado
  • ✅ Atualização de dados sem alterar senha
  • ✅ Registro de data da última alteração
  • ✅ Busca de usuários por nome
  • ✅ Garantia de unicidade de email
  • ✅ Validação de login obrigatória
  • ✅ Versionamento de API (/api/v1/)
  • ✅ Dois tipos de usuário (Cliente, Dono de Restaurante)
  • ✅ Padrão ProblemDetail (RFC 7807)
  • ✅ Documentação com Swagger
  • ✅ Banco de dados relacional (PostgreSQL)
  • ✅ Dockerização completa

2. Arquitetura da Aplicação

Visão Geral

A aplicação segue uma arquitetura em camadas (Layered Architecture) com separação clara de responsabilidades, facilitando manutenção, testes e evolução do código.

Diagrama de Camadas

API Request ↓ Controller (UsuarioController, AuthController) ↓ Service (UsuarioService, AuthService, TokenService) ↓ Repository (UsuarioRepository - Spring Data JPA) ↓ Database (PostgreSQL) ↓ Response (DTOs)

Componentes Principais

Camada Componentes Responsabilidade
Controller UsuarioController
AuthController		 Receber requisições HTTP, validar entrada, coordenar chamadas à service
Service UsuarioService
AuthService
TokenService		 Lógica de negócio, validações, orquestração de operações
Repository UsuarioRepository Acesso a dados, queries SQL abstratas
Security SecurityConfig
AuthFilter		 Configuração de segurança, autenticação JWT
Handler GlobalExceptionHandler Tratamento centralizado de exceções, respostas padronizadas
DTO *RequestDTO
*ResponseDTO
ProblemDetailDTO		 Transferência de dados entre camadas

Padrões de Projeto Utilizados

  • MVC: Separação Model-View-Controller
  • DTO: Data Transfer Objects para comunicação entre camadas
  • Repository: Abstração de acesso a dados
  • Service: Lógica de negócio centralizada
  • Exception Handler: Tratamento global de exceções
  • Mapper: Conversão entre entidades e DTOs

Princípios SOLID Aplicados

  • S (Single Responsibility): Cada classe tem uma única responsabilidade
  • O (Open/Closed): Aberto para extensão, fechado para modificação
  • L (Liskov Substitution): Interfaces bem definidas para substituição
  • I (Interface Segregation): Interfaces específicas e segregadas
  • D (Dependency Inversion): Injeção de dependências com Spring

Entidade: Usuario

Tabelausuarios

Campos

Campo Tipo Constraints Descrição
id BIGINT PK, Auto-increment Identificador único
nome VARCHAR(255) NOT NULL, 3-120 chars Nome completo do usuário
email VARCHAR(255) UNIQUE, NOT NULL, Valid Email Email único para login

Objeto Embarcado: Endereco

Tipo@Embeddable
RelacionamentoUm-para-Um (Embarcado)

Campos

  • rua: VARCHAR(120), obrigatório
  • numero: VARCHAR(10), obrigatório
  • cidade: VARCHAR(80), obrigatório
  • estado: VARCHAR(2), obrigatório (exatamente 2 caracteres)
  • cep: VARCHAR(10), obrigatório (formato: XXXXX-XXX ou XXXXXXX)

Enum: TipoUsuario

  • CLIENTE: Usuário cliente do restaurante
  • DONO_RESTAURANTE: Proprietário/gestor do restaurante

Relacionamentos

  • Usuario → Endereco (Um-para-Um, Embarcado)
  • Usuario implementa UserDetails (Spring Security Integration)

4. Endpoints da API

Base URL

http://localhost:8080/api/v1

Autenticação

POST /login
Descrição: Autentica usuário e retorna JWT
Body: { "email": "user@email.com", "senha": "password" }
Response: { "token": "jwt_token", "email": "user@email.com" }
Status: 200 OK ou 401 Unauthorized

Usuários - Criar

POST /usuarios
Descrição: Cria novo usuário
Body: UsuarioRequestDTO
Response: 201 Created
Erros: 409 Conflict (duplicado), 400 Bad Request (validação)

Usuários - Listar

GET /usuarios?page=1&size=10
Descrição: Lista usuários com paginação
Params: page (default: 1), size (default: 10)
Response: List<Usuario> - 200 OK

Usuários - Buscar por ID

GET /usuarios/{id}
Descrição: Retorna usuário específico
Response: Usuario - 200 OK
Erros: 404 Not Found

Usuários - Buscar por Nome

GET /usuarios/search/nome?nome=value
Descrição: Busca usuários por nome (sem exposição de senha)
Response: List<UsuarioUpdateDTO> - 200 OK

Usuários - Atualizar (sem senha)

PUT /usuarios/{id}
Descrição: Atualiza dados do usuário (não altera senha)
Body: UsuarioUpdateDTO
Response: Usuario - 200 OK
Erros: 404 Not Found, 409 Conflict, 400 Bad Request

Usuários - Trocar Senha

PATCH /usuarios/{id}/senha
Descrição: Altera senha do usuário
Body: TrocarSenhaRequestDTO
Response: 204 No Content
Erros: 404 Not Found, 400 Bad Request (senha incorreta)

Usuários - Deletar

DELETE /usuarios/{id}
Descrição: Deleta usuário
Response: 204 No Content
Erros: 404 Not Found

Resumo de Endpoints

Método Endpoint Descrição Auth
POST /login Autenticar ❌
POST /usuarios Criar ❌
GET /usuarios Listar ❌
GET /usuarios/{id} Buscar ID ❌
GET /usuarios/search/nome Buscar Nome ❌
PUT /usuarios/{id} Atualizar ❌
PATCH /usuarios/{id}/senha Trocar Senha ❌
DELETE /usuarios/{id} Deletar ❌

6. Coleção Postman

Arquivo

Cobertura de Testes

A coleção contém 24 testes organizados em 7 grupos:

Grupo Testes Cenários
Autenticação 3 Login válido, email inválido, senha incorreta
Criar Usuários 9 Válido, duplicado, inválido (email/CEP/estado/campos)
Listar Usuários 3 Todos, page 1, page 2
Buscar Usuários 4 Por ID (existente/inexistente), por nome
Atualizar Usuários 4 Sucesso, ID inexistente, duplicado, validação falha

  3. Selecione o arquivo "Gestao_API_Postman.json"
  4. Configure a variável "baseUrl" para http://localhost:8080
  5. Execute os testes na sequência sugerida

7. Banco de Dados

Tecnologia

  • Engine: PostgreSQL 12+
  • Driver: org.postgresql:postgresql (runtime)
  • ORM: Spring Data JPA com Hibernate
  • Migrations: Flyway

Configuração de Conexão

URL: jdbc:postgresql://localhost:5432/mydatabase Usuário: myuser Senha: secret Driver: org.postgresql.Driver

DDL - Tabela usuarios

CREATE TABLE usuarios ( id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY, nome VARCHAR(255), email VARCHAR(255) UNIQUE, login_username VARCHAR(255) UNIQUE, senha VARCHAR(255) NOT NULL, tipo_usuario VARCHAR(255) CHECK (tipo_usuario IN ('CLIENTE','DONO_RESTAURANTE')), rua VARCHAR(255), numero VARCHAR(255), cidade VARCHAR(255), estado VARCHAR(255), cep VARCHAR(255), data_ultima_alteracao TIMESTAMP(6), PRIMARY KEY (id) );

Constraints e Índices

  • PRIMARY KEY: id (auto-increment BIGINT)
  • UNIQUE: email, login_username
  • NOT NULL: senha, tipo_usuario
  • CHECK: tipo_usuario IN ('CLIENTE','DONO_RESTAURANTE')
  • Índices implícitos: PK e UKs automaticamente indexadas

Migrações Flyway

Versão Arquivo Descrição
1 V1__Create_Usuario_Table.sql Criação inicial da tabela usuarios

Estratégia de Validação

A aplicação utiliza ddl-auto=validate no Hibernate:

  • Valida se o schema existe
  • Não cria ou modifica tabelas automaticamente
  • Falha na inicialização se houver inconsistências
  • Migrations são gerenciadas exclusivamente pelo Flyway

8. Execução com Docker

Arquivo: compose.yaml

services: postgres: image: 'postgres:latest' container_name: postgres_db environment: - 'POSTGRES_DB=mydatabase' - 'POSTGRES_PASSWORD=secret' - 'POSTGRES_USER=myuser' ports: - '5432:5432'

Pré-requisitos

  • Docker 20.10+
  • Docker Compose 1.29+
  • Java 17+
  • Maven 3.8+

Instruções de Execução

Passo 1: Iniciar Banco de Dados

cd tech_challenge_fase1 docker compose up -d

Verifica: Docker Desktop rodando, pode demorar alguns segundos para o PostgreSQL inicializar.

Passo 2: Compilar a Aplicação

./mvnw.cmd clean compile

Isso baixa dependências Maven e compila o código fonte.

Passo 3: Executar a Aplicação

./mvnw.cmd spring-boot:run

A aplicação inicia na porta 8080. Espere mensagens de sucesso.

Passo 4: Testar Endpoints

# Terminal novo curl -X GET http://localhost:8080/api/v1/usuarios

Passo 5: Acessar Documentação

  • Swagger UI: http://localhost:8080/swagger-ui.html
  • OpenAPI JSON: http://localhost:8080/v3/api-docs

Parando a Aplicação

# Parar a aplicação Spring Boot: Ctrl+C no terminal # Parar o PostgreSQL: docker compose down # Parar e remover volumes: docker compose down -v

Troubleshooting

  • Porta 5432 em uso: docker ps, identifique container, docker stop {id}
  • Conexão recusada: docker compose logs postgres
  • Erro de compilação: ./mvnw.cmd clean (limpa cache Maven)
  • Porta 8080 em uso: Altere spring.server.port em application.properties
